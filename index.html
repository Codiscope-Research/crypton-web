What is Crypton?

Crypton is a framework for building cryptographically secure cloud applications. Such applications offer meaningful privacy assurance to end users because the servers running the application cannot read the data created and stored by the application.

Using Crypton, developers can for the first time easily build featureful, multi user collaborative applications that offer total privacy and security "out of the box." Crypton transparently does all the hard parts of cryptography behind the scenes. This allows developers to focus on domain specific challenges.

How does this change the landscape?

Users will no longer have to chose between the convienience of the cloud applications and protecting their privacy.

Today there are few choices for effectively encrypted cloud storage. There are even fewer choices for feature rich and collaborative encrypted cloud apps. In the future there will be applications like Evernote, Basecamp, Freshbooks, and Google Docs and many others with meaningful guarantees about your privacy.

Crypton allows developers to provide customers a truly private storage and collaboration environment with no access to customer data, without having to rely on 3rd party security layers or post development hacks.

Who is it for?

Crypton is the perfect fit for any developer wishing to provide end to end security and privacy without the hassle of securing a inherently open environment such as Dropbox API or Amazon S3. With Crypton everything is meaningfully encrypted from day one.

Why is it interesting?

More people are becoming 'privacy aware." Enterprises refuse to adopt solutions where the developer/service provider can access critical internal data. Crypton is the first application framework that provides a foundation for building zero-knowledge cloud products.

With Crypton you can develop secure and private applications aimed at the already large and growing audience of 'privacy aware' customers without the added burdens of writing cryptography, scaling databases, or matching a cloud API with 3rd party security solutions.

Has this been done before?

To our knowledge there is no existing framework that handles all the encyrption, database storage, and private user to user communication needed to build a zero knowledge cloud application.

Other cloud applications have been created that involve cryptography, but not in a generalized, reusable form that everyday developers could easily use to build a wide range of new apps.

Should this be happening at all? Isn't this a bad idea, fraught with peril? Crypto code is hard and Javascript Crypto is doomed!

Building Crypto code is indeed famously hard. We have some experience with cryptography, but most importantly we have an understanding that it needs auditing and verification by experts, and a development process that lends itself to this review.

It's better that the hard crypto code happens here in a centralized and general purpose application framework as an open source project where we can all work together, rather than requiring developers to either write their own crypto code, or just avoid building applications with meaningful end user privacy assurance.

When building an application with Crypton, the cryptography is entirely encapsulated and hidden. You're just using a simple interface to data storage.

There are additional concerns and arguments about "Browser based" crypto.

First, we need to be clear that Javascript and "browser based" are not always the same thing. Javascript code might be a HTML5 mobile app running on Android, or a NodeJS based desktop app created with something like AppJS. Each of these scenarios are ideal environments for deploying Crypton applications and sidestep the objections to the browser environment.

For browser based applications there are some mitigations available. Chrome includes a native secure random number generator. Firefox is working on it . We'll do what we can to avoid data exposure through side channels, such as using web workers where possible to avoid leaking timing information into the event loop.

Finally, remember the threat model we're protecting against. Specifically we're protecting the service operators from the dangers of having readable database of end user data. Most of the arguments against browser based crypto don't apply to this goal.

What types of applications should be built with Crypton?

The most appropriate applications are those with significant personal user generated content.

For example 37signals Basecamp, a collaborative project management application, would be a good candidate. Basecamp is an interface for organizing TODOs, tracking your time, collecting documents, assets, timelines, calendars, and general communication among a group of people working on a specific project. All of the "data" that goes into the application is provided by the users. The software itself is generic — it can work with any combination of documents, calendar events, and so on.

Most importantly, the servers' administrator don't profit from seeing this data, and even get into trouble when looking at it!

Many applications will have a mix of content created by users and data supplied by the server, and may use Crypton for the sensitive stuff and traditional database storage for community data. For example, a site like Reddit might only use Crypton for only a few features, such as the private messaging system where users send messages directly from one account to another, or a user's own list of saved posts and comments.

In general, you can think of the most successful applications in any category, and ask yourself if the market has room for a competitor with meaningful end user privacy assurance? How about Evernote with truly private notes that you could privately share with just the people you chose? RescueTime? Hipchat? Skype? Firebase? Flickr? Google Docs? FreshBooks?

What types of applications wouldn't work well with Crypton?

It wouldn't make much sense to build Netflix with Crypton. There's no opportunity for user data to be secret from the server. All the "data" in the system is essentially data created by the server -- the directory of movies and shows available for viewing, and the viewing history of individual users. The server necessarily knows which movies each user watches because it must stream them to that user. The recommendation system is improved by direct calculations made from visible user history.

Why should application developers care about end users' privacy?

Because having such data can get you into trouble. http://www.nytimes.com/2013/02/17/technology/if-youre-collecting-our-data-you-ought-to-protect-it.html?_r=1&

Because privacy is a feature you can sell. If you had the choice between using an app where your data was exposed, and an app that guaranteed your privacy, which would you chose?

Because withholding privacy allows new competitors to gain a market toehold and become entrenched. For example, if a competitor to Evernote emerged tomorrow which offered a subset of functionality but true privacy, some portion of the market would chose that competitor over Evernote.

No matter how many additional innovations Evernote added, some portion of the market would continue to chose the private solution. This allows the competitor a guaranteed customer base and revenue stream, allowing it become progressively more competitive over time.

Companies offering a freemium product should be especially concerned. In the above example, end users are likely to chose the free Evernote for their less sensitive and public stuff while paying the competitor to protect their most important notes!

Is Crypton Horizontally Scalable for a very large, high performance site with hundreds of millions of users?

Yes, and far easily than a site built with traditional approaches to data storage.

The backend data structures are designed for sharding as a site grows beyond the capabilities of a single cluster of database servers. This means the process of building a very large Crypton app isn't about "scaling it to support 100 million users." It's more like: "scale it to support 1 million users" and then repeat that 100 times. Scaling out rather than up -- a far easier challenge.

The nature of client side rather than server side intelligence naturally means that the server never has to perform joins between containers. The server infrastructure gets out of the business of doing calculations on data. Its job is just to reliably store and retrieve data while enforcing access policies and protocols.

How is Crypton's reference implementation built?

Crypto code is hard. What code review and auditing has the cryptography in Crypton's specification and implementations received? How will this continue moving forward?

Crypto code is isolated into particular designated modules of the framework. SpiderOak senior engineers will do review of all changes to those modules before each major and minor release.

SpiderOak will purchase expert code review from outside security specialists before each major release that involves changes to the crypto code. The crypto portions of the code won't typically change between minor versions.

This is sustainable. Over time, the majority of the framework's code base won't be the crypto code itself, but rather database features and other higher level functionality built on top of the crypto layer.

What testing, code quality and development guidelines are in place to make the code accessible to reviewers and bug hunters?

Building software involves bugs, and bugs in crypto code are often fatal. Research has shown that a variety of quality control tools are effective at finding different types of bugs, and consequently the combination of several techniques is most likely to lead to reliable software. Among these, human code review is probably the most important.

Crypton is written with human review and testability in mind. This means: consistent style, single purpose, simple functions, clear code separation, meaningful function and variable names, and careful code organization. Data structures are specified independently of the code that implements them.

Additionally, Crypton uses all of the following:

  • Static Analysis

    All of the Crypton code created by us passes jshint with a custom configuration that's fairly strict and disallows most of the problematic areas of JS.

  • Standard Style Guide w/ Code Complexity Controls
  • Unit Tests
  • Integration Tests
  • PhantomJS and Node Integration
  • Continuous Integration
  • Code Review
  • Intentional End User QA by trained UI breakers

What additional capabilities and enhancements will be added as the project matures?

  • Client: Native Browser Plugins for the slow and sensitive crypto ops
  • Backend: Application level data partitioning
  • Websocket and long polling support, for realtime notifications for changes
  • Backend: Faster implementation of realtime transient messaging
  • Backend: Support for pluggable external binary storage: Nimbus.io, Amazon S3, etc.
  • Object DB: Pluggable User Created Adapters and Differs of more complex data structures, allowing faster container refresh and smaller data storage.
  • Object DB: More Standard Adapters and Differs
  • Object DB: Container Triggers: Automatically run code on every save. For example, automatically maintain a searchable index by keyword in separate container(s)
  • Object DB: Container Compaction
  • Object DB: More aggressive container caching for faster application load times and interactivity.
  • Object DB: Specialized containers that implement more efficient maps for strings and large integers.
